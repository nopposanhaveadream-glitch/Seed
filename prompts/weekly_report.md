# Seed0 フェーズC 定期レポート依頼

フェーズC（Phase 1長期観察、4/19〜5月上旬）中の定期レポートをお願いします。

## 出力先

`~/.seed0/reports/phase_c_report_YYYY-MM-DD.md` として保存してください。

## レポート内容

### 1. 基本メトリクス（全レポート共通）

- 稼働時間（時間 / 日数換算）
- 現在のPID、プロセス状態
- Q学習: 状態数 / エントリ数 / ε値
- 記憶件数（STM / LTM / 合計 / 最大値 / 最小値）
- VE=0比率（日別・全期間平均）
- 活動率（行動している時間 / 全時間）
- 行動比率（各actionの選択回数と割合）

### 2. Q値時系列（今回の重点観察項目）

**VE<30 状態における rest と sense_body のQ値推移を必ず含めてください。**

背景: フェーズA〜Bで「苦しいときほど感じようとする」現象（VE低下時にrest代わりにsense_bodyを選ぶパターン）が観察された。これが創発的な探索行動なのか、それともrestのQ値が育っていないだけなのかを切り分けたい。

出力フォーマット:

```
## VE<30状態でのQ値推移

期間: [レポート作成日から遡って直近7日]

| 日付 | rest Q値 | sense_body Q値 | 差分(sense_body - rest) |
|------|----------|----------------|------------------------|
| ...  | ...      | ...            | ...                    |
```

追加で、以下の観点で1〜3行のコメント:

- restのQ値は上昇傾向か、横ばいか、下降か
- sense_bodyがrestを上回り続けているか、クロスしたか
- もしクロスしたら、いつクロスしたか

### 3. Q状態数の伸び率（天井判定用）

**目的**: 環境1の情報量をSeed0が消化しきったかの判定。

**判定基準**: 7日間連続で状態数の増加が全体の3%以下に収まったら「天井到達」と判定する。

出力フォーマット:

```
## Q状態数の伸び率

| 日付 | 状態数 | 前日比増加 | 7日前比増加率 |
|------|--------|-----------|--------------|
| ...  | ...    | +N        | +X.X%        |

判定: [天井未到達 / 天井到達（YYYY-MM-DD時点）]
```

### 4. その他の注目点（フェーズCで観察中の項目）

以下を各1〜3行で報告:

- purge_memoryのQ値推移（memory=mid での値が上がっているか）
- 記憶件数のピーク値（300件超の暴走が抑えられているか）
- VE=0比率の傾き（4/15のブレークスルー後の安定性）
- **user_idle_seconds のbaseline収束状況**: 平均、分散、日中/夜間の値の分布(二峰性の有無)。注意: user_idle_secondsは現状、報酬計算にも状態離散化にも組み込まれていない。そのため行動パターンへの直接的な影響は構造上出ない。ここで見るのは「関係代謝を組み込むための値域調査」であり、行動差の発生ではない。
- 活動率の推移（restを選ばない傾向が持続するか）

### 5. 異常・気になる兆候

以下のいずれかに該当する場合は明示的に報告:

- VE=0比率が前日比で2倍以上に増加
- 記憶件数が300件を超えた
- 強制睡眠（疲労≥95）が発生した
- プロセスが停止した / 再起動が必要になった
- 想定外のエラーログ
- Q学習が特定の行動に偏りすぎている（特定actionが全体の70%超）

## 重要な制約

**このレポートは観察のみが目的です。介入・修正の提案は不要です。**

第一原則「構造だけ与え、動機は構造から生まれる」に従い、フェーズC中はパラメータを一切変更しません。気になる傾向があっても、報告のみでお願いします。設計判断は5月上旬の全期間振り返りで行います。
