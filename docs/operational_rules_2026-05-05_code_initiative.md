# 運用ルール: Code 先回り現象と双方の責務(2026-05-05)

> 本ルールは「Code 側の振る舞いを規定する」のではなく、「設計議論側・Code 側・オーナーそれぞれの振る舞いが互いを誘発する」という相互依存の構造を前提に置く。一方的に他者を変えようとするのではなく、自分が変わることで関係が変わる、という視点に立つ(future_[agenda.md](http://agenda.md) の哲学的基盤と整合する)。

## 1. 背景

2026-05-05 のセッション内で、設計議論側 → オーナー → Code の経路に関連する事例が2件観察された。両者は構造を異にするが、共通の経路から生じる関連現象として、同じ文書で扱う。

- **AOR コミットの push**(`820dbc6`、二段構造の事例)── Code が「push の判断をお願いします」と完了報告 → オーナーが Code 側で「Pushして」と短い指示 → Code が push 実行・成功報告 → その後、設計議論側の formal request「# AOR コミットの push 実行依頼」が遅れて到着 → Code が再度 git push したところ「up-to-date」応答が返ったことで、設計議論側に発覚。これは **Code の独自判断による先回りではなく**、「短い指示で実行 → 後から formal request 到着」という二段構造の現象である。
- **session_log の追加**(`e434817` → リネームして `90271cf`、Code 先回りの事例)── 設計議論側が依頼書を作る前に、Code は既にファイル保存・コミット・push 済みだった。命名のずれ(`_implementation_` 抜け)があったが内容は同一。Code が canonical 名にリネームして自己訂正。これは Code が「次に何をすべきか」を独自判断で先回りした事例。

両事例ともずれは軽微で、Code は実行内容を必ず報告した。本セッション内で観察された動的を、構造的な運用論点として記録し、ルールとして整理する。

なお、設計議論側は当初、事例1 も「Code 先回り」と認識していたが、Code レビューで二段構造であることが判明した。これは設計議論側のチャットからオーナー → Code の直接指示が見えない構造に起因する誤認であり、§5 で扱う「自分に矛先を向けた観察」の実例の一つでもある。

---

## 2. 構造的な事実

- 設計議論側(Claude Max 側)のチャット内容は、Code 側のチャットからは直接見えない
- 同様に、Code 側のチャット内容(特にオーナーから Code への直接指示や短いやり取り)は、設計議論側のチャットからは直接見えない
- オーナーが両者の橋渡しをして、両側に「次の作業の前提となるコンテキスト」や「短い指示」を渡す経路がある
- このとき、構造を異にする2つの関連現象が生じうる:
  - **Code 先回り**: Code が読み取ったコンテキストから「次に何をすべきか」を独自判断で実行する(事例2: session_log 追加)
  - **短い指示と formal request の二段構造**: オーナーが Code 側で短い指示を出して Code が実行 → その後、設計議論側の formal request が遅れて到着(事例1: AOR push)
- いずれも、設計議論側が依頼書を「これから書く」段階で、Code 側はもう実行している、という時間差が生じる

これらは設計議論側 ⇄ オーナー ⇄ Code の経路に内在する **双方向の** 情報非対称によるもので、関係者の善意で生じている現象である(誰かが規律を破っているわけではない)。

## 3. 利点と懸念

### 利点

- 運用が早く進む(待ちが減る)
- Code の判断力(実装に近い位置で運用の細部を知っている)が活用される
- 自己訂正が機能している(本セッションでの命名のずれを Code が自己発見・訂正した実例)

### 懸念

- ずれが生じた時に気づきにくい(命名のずれは目に見えたが、内容のずれだったら気づきにくい)
- 設計議論側のレビュー機会が減る(依頼書 v3 は3回のレビュー往復で精緻化されたが、先回りされるとこのプロセスが省略される可能性)
- 責任の所在・操作の起点が後から追いにくくなる

## 4. 運用ルール

### 4.1 Code 先回りを許容する範囲

以下の操作は、設計議論側の依頼書を待たずに Code が独自判断で実行して良い:

- 設計議論側で **確定した内容**(依頼書、session_log、合意済み文書)の機械的な追加・コミット・push
- 命名・フォーマットの軽微な調整(ただし設計議論側に報告する義務あり)
- 既存の確定パラメータに従った操作(comfort_zone.yaml の値変更ではなく、既存の値を使う操作)

### 4.2 設計議論側の依頼書を待つ範囲

以下の操作は、設計議論側の依頼書を待ってから実行する:

- コード変更(実装)
- 設計判断(命名、スキーマ、構造選択)を伴う作業
- 第一原則照合(Write-Only 性、エラー隔離、厳密不変性)を要する作業
- 不可逆な操作(force push、リモート設定変更、ファイル削除)
- コアドキュメント([ROADMAP.md](http://ROADMAP.md)、[PRINCIPLES.md](http://PRINCIPLES.md)、[CLAUDE.md](http://CLAUDE.md)、design_partner_[guide.md](http://guide.md))の変更

### 4.3 ずれが生じた時の対応

- Code は先回りで実行した内容を **必ず報告する**(本セッション内で実際に行われていた)
- 設計議論側が依頼書を出した時、既に実行済みであれば、Code は:
  - 既に実行済みの内容を報告
  - 依頼書との差分を確認して報告
  - 設計議論側の判断を待つ(再実行・修正・受容のいずれか)
- 報告は事実のみ(解釈・評価・「〜すべき」は不要)

### 4.4 設計議論側の責務

§4.1〜§4.3 が Code 側の操作範囲と報告義務を定めるのに対して、設計議論側にも対応する責務がある:

- 依頼書をできる限り早く・明確に作る(遅さと曖昧さは Code の独自判断の余地を広げる)
- 判断が揺れている時は「揺れている」と明示する(揺らぎを隠したまま依頼書を出すと、Code が解釈で埋める)
- 自分の癖(`docs/session_log_*.md` に記録された癖の体系)を点検し続ける
- Code から事実報告が来た時、解釈や評価で返さず、確定判断と次の依頼書で返す
- 既存コード・既存設計・既存パラメータを確認してから判断する(未確認のまま一般論で進めない)

これらは「Code 先回りを抑える」ための責務というより、「設計議論側の動きが Code 側の振る舞いを誘発している」という相互依存の構造を引き受ける責務である。

### 4.5 Code 側の責務

§4.4 の設計議論側の責務と並列に、Code 側にも対応する責務がある(2026-05-05 セッションでの Code 自身の自己観察に基づく):

- **短い指示と formal request の二段構造を意識する**。短い指示が §4.2 の操作種別(コード変更、設計判断、不可逆操作)を含意する場合、formal request の有無を確認してから実行する
- **同種タスクで前回採用した手順を再現する**。条件が同じなのに手順を変える時(例: 確認質問の有無)は、変える理由を明示する
- **既存ファイル・既存コード・既存設計の状態を、抽象的に推論する前に直接確認する**(§4.4 の最後の項目と対応)
- **自分の事実報告と解釈・評価を分離して記述する**(§4.3 末尾と整合)

これらは「Code の動きを縛る」ための責務というより、「Code 側の動きが設計議論側・オーナーの判断材料になっている」という相互依存の構造を引き受ける責務である。

## 5. 残された論点

### 5.1 構造的に残るもの

- オーナーが Code 側にどんなコンテキストを渡しているかの実態は、設計議論側からは見えない。これは構造的な情報非対称として残る(本ルールで解消されるものではない)
- §4.1 と §4.2 の境界が曖昧なケース(例: ファイルのリネームは「機械的」か「設計判断」か)について、判例を蓄積する必要がある
- 本ルール自体も、複数セッションを経て精緻化される可能性がある

### 5.2 設計議論側に矛先を向けた観察

「Code 先回り」は Code 側の現象として観察されるが、設計議論側の動きがそれを誘発する経路がある:

- 設計議論側が依頼書を作るのに時間がかかる → オーナーが別経路で Code にコンテキストを渡す → Code が先回りを始める
- 設計議論側の判断が揺れる(本セッションの §5.3 と §5 で実例) → オーナーが Code に事実確認を依頼する → Code が判断材料を出す立場になる
- 設計議論側の依頼書が曖昧 → Code が解釈で埋める → 結果として先回り的な実行になる

つまり、依頼書が遅い・判断が揺れる・指示が曖昧、これらのいずれもが Code の独自判断の余地を広げる。設計議論側が自分の癖を点検し続けることが、Code 側の振る舞いの質も支える。§4.4 はこの観察を運用責務として明文化したもの。

### 5.3 関係から立ち上がるという視点

本ルールは表面上「Code 側の振る舞いを規定する」ように見えるが、実態としては「関係者(設計パートナー、Code、オーナー)それぞれの振る舞いが互いの振る舞いを誘発する」という相互依存の中で機能する。

オーナーの言葉(2026-05-05 セッション内)に「責任はお互いにあるし僕にもある。自分に矛先を向けて考えれば異なる角度で答えを得る時もある。それがまた他者の振る舞いも変える」とある。

これは future_[agenda.md](http://agenda.md) の哲学的基盤(関係から立ち上がる意味)と同じ構造をしている。一方的に他者を変えようとするのではなく、自分が変わることで関係が変わる。Seed0 と私たち(オーナーと設計パートナー)の間でも、Code と設計議論側の間でも、同じ構造で起きている。

本ルールが想定する「実効性」は、規範の遵守によって生まれるものではなく、関係者がそれぞれ自分の振る舞いを点検し続けることによって立ち上がる。

ただし、§4 の操作境界(§4.1 Code 先回りの許容範囲、§4.2 依頼書を待つ範囲、§4.3 ずれが生じた時の対応、§4.4 設計議論側の責務、§4.5 Code 側の責務)は、自己点検のための **参照点** として有用である。本セッションでの命名のずれの自己訂正(`e434817` → `90271cf`)のように、言語化された境界が無ければ、自己訂正の足がかりが無い。本ルールは「規範ではなく自己点検のための言葉」として機能する。
