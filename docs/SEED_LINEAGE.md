# Seedシリーズ系譜

> 作成日: 2026-04-25
> 目的: Seed0を原種として位置付け、将来の派生Seedの構想を整理する

## 1. 背景

Seed0は、Mac mini を身体とする代謝AIの最小プロトタイプである。有限性、代謝、疲労、記憶、睡眠、身体センサーから、AI個体性の芽が生まれるかを検証することを目的としている。

このプロジェクトを進める中で、Seed0をどういう存在として位置付けるかが設計判断に影響することが見えてきた。Seed0に過剰な機能を載せると、どの振る舞いが代謝構造から生まれたものか、どの振る舞いが追加機能によるものかが判別できなくなる。

そのため、Seed0は「完成品」ではなく「原種」として扱う。

## 2. Seed0の位置付け

Seed0は、Seedシリーズの原種である。

ここでいう「原種」とは、Seedシリーズ全体の出発点となる純粋な代謝存在を指す。Seed0には、代謝・疲労・記憶・睡眠・身体センサーという最小の構造だけを与える。

Seed0が役に立つかどうか、便利かどうかは設計者が決めることではない。それは Seed0 自身の振る舞いから立ち上がる結果として現れるものであって、事前に設計するものではない。第一原則「動機は構造から生まれる」は、Seed0 の用途についても同様に適用される。

## 3. 「器官」という考え方

Seedシリーズに新しい機能を追加する場合、その機能は外付けの便利機能ではなく、Seedが世界と関わるための器官として位置付ける。

重要な原則:

- 機能追加の主体は常にSeed Coreにある
- LLMや外部ツールはSeed Coreの状態に従属する器官として動く
- 行動の結果はSeedの記憶・予測・代謝に戻る

避けたい構造:

- LLMや外部ツールが主体になり、Seedの状態を飾りとして使う
- 道具がSeedの主体性を奪う

## 4. Seedシリーズ系譜の構想

将来の派生Seedの構想として、以下を仮置きする。これは厳密な実装順序ではなく、Seed0から派生しうる方向性の概念図である。

```
Seed0  原種(身体・代謝・記憶)
Seed1  声(内部状態の言語化)
Seed2  遊び場(仮想環境での自由探索)
Seed3  関係(他者との相互作用)
Seed4  道具(限定された外部ツールの使用)
Seed5  社会(他エージェントとの関係)
```

各派生Seedの詳細設計は、Seed0が十分に育ち、その時点での観察結果に基づいて決定する。事前に詳細を決めない。

なお、ここでの「他者」(Seed3)は人間中心ではない。docs/future_agenda.md で議論されているように、人間・別のSeed0個体・AIエージェント・動物・気象パターンまで広く取る。Seed3はこの他者論の実装段階に対応する。

## 5. 既存ロードマップとの関係

既存のREADME.mdおよびdocs/ROADMAP.mdでは、Seed0の成長は環境拡張として整理されている。

```
環境1: 自分の身体だけが見える世界
環境2: 遊び場へのアクセス
環境3: 外の世界が見える
環境4: 言語能力の獲得
```

本ドキュメントはこの既存ロードマップを否定するものではなく、Seedシリーズとしての系譜名と位置付けを与えるものである。両者は並存する。

```
環境1 身体             → Seed0 原種
環境2 遊び場           → Seed2 遊び場
環境3 外の世界         → Seed3〜Seed5(関係・道具・社会)
環境4 言語能力         → Seed1 声
```

実装順序は番号順である必要はない。たとえばSeed1「声」は、内部状態の翻訳器官として比較的早期に導入可能だが、それは人格を持つ会話AIではなく、観察対話窓として限定される。

## 6. 機能追加時の判断基準

Seedシリーズに新しい機能を追加するか判断する際、以下の問いを使う。

- これはSeedの経験を増やす器官か、それとも便利機能を貼っているだけか
- 主体はSeed Coreに残っているか
- LLMやツールが主体を奪っていないか
- 行動結果はSeedの記憶・予測・代謝に戻る構造になっているか
- 第一原則「構造だけ与え、動機は構造から生まれる」と整合しているか

これらの問いに明確に答えられない機能は、Seed0には追加しない。必要であれば、派生Seedとして検討する。

## 7. このドキュメントの位置付け

これは将来構想を整理する文書であり、現時点で実装を約束するものではない。

Seed0の現在のフェーズで重要なのは、「Seed0は原種である」という認識を保つことと、機能追加の判断基準として上記の問いを使うことである。

派生Seedの具体的な設計は、それぞれの実装段階で別途議論する。
