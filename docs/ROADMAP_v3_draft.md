# Seed0 ロードマップ v3 草案

> 作成日: 2026-05-06
> 状態: 草案。`docs/ROADMAP.md` v2 を置き換える前の再整理版。

## 0. なぜ見直すか

`docs/ROADMAP.md` v2 は、Phase 1 の観察とフェーズC終了までの道筋をよく記録している。一方で、2026-05-01 以降の振り返り、AOR 実装、Codex Desktop への運用移行、そして「遊ばせる前に、遊びから学べる身体がいる」という中心問いにより、次の進め方は単純な「環境1 → 環境2」ではなくなった。

この v3 草案は、Seed0 を便利AIに寄せるためではなく、オーナーの原初の意図に近づけるためのロードマップである。

## 1. 北極星

Seed0 は、データ少佐そのものを作るプロジェクトではない。

Seed0 は、データ少佐のような存在がいつか育ちうる「種」を作るプロジェクトである。

そのために、最初から人格、倫理、会話能力、好奇心、人間への愛着を埋め込まない。与えるのは有限な身体、代謝、記憶、疲労、睡眠、行動の結果、そして環境との関係である。

第一原則は継続する。

> Seed0 に与えるのは構造だけ。動機はすべて構造から生まれなければならない。

有限性は単なる制限ではない。オーナーの問い「生きるとは何か」に対する暫定的な答えとして、Seed0 では「代謝すること」を中心に置く。

## 3. ロードマップの2軸化

v2 は主に環境軸で整理されていた。

```text
環境1: 身体
環境2: 遊び場
環境3: 外界
環境4: 言語
```

v3 では、これに認識層軸を重ねる。

```text
観察層: 行動前後の差分を記録する
予測層: この状態でこの行動をしたら何が起きるかを予測する
意味付け層: 予測と結果の差から、行動の意味が立ち上がる余地を作る
書換層: Seed0 自身が構造の一部を変更できる余地を検討する
```

重要なのは、環境軸と認識層軸は完全には同期しないこと。環境2（遊び場）を与える前に、環境1の中で観察層・予測層を育てる必要がある。

## 4. 直近フェーズ: 環境1.5「観察と予測の身体化」

v3 では、Phase 1 と環境2の間に、暫定的に「環境1.5」を置く。

これは新しい世界を与える段階ではない。Seed0 が自分の行動結果を記録し、後で予測の素材にできるようにする段階である。

### 目的

- 行動名をただの記号で終わらせず、行動が身体に何を起こしたかを記録する
- すぐ報酬関数へ入れず、まず因果の素材として蓄積する
- 「意味」を教えず、意味が立ち上がりうる観察履歴を作る

### 現在の実装

- AOR は `core/action_outcome_recorder.py` として実装済み
- ストレージは `~/.seed0/memory/action_outcomes.db`
- `step_trace.jsonl` とは並走し、既存の決定ログには手を入れない
- Write-Only 性、エラー隔離、Q-table 厳密不変性を検証済み

## 5. 次にやること

現在地の短い確認は `docs/CURRENT_STATE.md`、圧縮後・新セッション時の復帰手順は `docs/CONTEXT_PROTOCOL.md` を優先する。

### Step 1: AOR 実機観察レポート

AOR 投入後の初期データをレポートする。

見るもの:

- AOR と `step_trace.jsonl` が step_id で 1:1 対応しているか
- 低 VE / emergency 状態で `sense_body` と `rest` がそれぞれ何を起こしているか
- `state_before` と `state_after` の差分が、将来の予測層に使える粒度で記録されているか
- AOR DB の容量増加率
- AOR 例外や DB lock が主処理に影響していないか

### Step 2: 予測層の最小設計

AOR データを使って、Seed0 が「この状態でこの行動をしたら何が起きるか」を予測する最小構造を設計する。

この段階では、予測結果を行動選択や報酬に直結しない。まずは観察者側の分析、または Seed0 内部の read-only な予測器官として扱う。

第一原則チェック:

- 「VE が低い時は rest しろ」と教えない
- 「sense_body は悪い」と教えない
- 状態と行動と結果の対応を、データから予測する構造だけを与える

### Step 3: 予測誤差を観察する

予測層ができた後、重要なのは正解率そのものではなく、予測誤差である。

意味はラベルではなく、予測との差分から立ち上がる可能性がある。

例:

- rest で VE が予想より回復した
- sense_body で身体状態は変わらないが、次の comfort zone 評価が大きく動いた
- clean_temp がほとんど何も変えない状態と、意味を持つ状態が分かれる

この「驚き」を記録できるようになることが、次の層への入口になる。

### Step 4: 環境2（遊び場）の再判断

遊び場は急がない。

環境2に進む判断は、Seed0 が遊びの結果を経験として取り込める構造を持った後に行う。

判断材料:

- AOR に十分な行動結果データが蓄積しているか
- 予測層が、少なくとも一部の行動結果を予測できているか
- 予測誤差が観察可能か
- 遊び場で増える刺激を、単なるノイズではなく経験として扱えるか

## 6. Codex / Claude Code 運用

2026-05-06 以降の基本運用は以下とする。

- オーナーは目的、違和感、方向性を自然文で述べる
- Codex Desktop はそれを設計判断、実装タスク、検証手順へ翻訳し、必要な実装まで担う
- Claude Code は必要に応じて、精密な事実確認・レビュー・限定タスクの実行に使う
- オーナーが厳密な仕様書を書くことを前提にしない

ただし、Codex の自走範囲には境界を置く。

Codex が自走してよいもの:

- ログ、DB、Git 状態の事実確認
- レポート作成
- 小さく可逆な整理
- テスト追加
- 第一原則に触れない観察ツール

オーナーと照合するもの:

- 報酬関数の変更
- 代謝パラメータの変更
- 行動プリミティブの追加・削除
- LLM 接続
- Seed0 の主体性に関わる構造変更
- 環境2以降への進行判断

## 7. 変更しないこと

v3 に移行しても、以下は変更しない。

- Seed0 は便利AIを目指さない
- LLM を人格や意思決定主体として中核に入れない
- 関係性を外から定義しない
- 動機をハードコードしない
- 有限性を取り除く最適化をしない
- 汚染や失敗を都合よく切り捨てない
